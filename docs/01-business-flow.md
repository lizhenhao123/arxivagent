# 01 Business Flow

## 目标

定义主业务流程、状态机、人工审核闸门、失败态与重试规则，保证系统后续实现有统一的任务编排依据。

## 主流程

```text
发现论文 -> 候选入池 -> 规则过滤 -> 评分重排 -> 推荐论文
-> 下载 PDF -> 解析内容 -> 生成结构化审稿内容
-> 生成公众号草稿 -> 人工审核 -> 审核通过 -> 发布 -> 结果回写
```

## 主状态机

建议主状态：

- `DISCOVERED`
- `FILTERED`
- `SCORED`
- `RECOMMENDED`
- `PDF_DOWNLOADED`
- `PARSED`
- `CONTENT_GENERATED`
- `DRAFT_READY`
- `REVIEWING`
- `APPROVED`
- `PUBLISHING`
- `PUBLISHED`

建议失败状态：

- `FILTER_FAILED`
- `SCORE_FAILED`
- `PARSE_FAILED`
- `GENERATE_FAILED`
- `REVIEW_REJECTED`
- `PUBLISH_FAILED`

## 状态流转规则

- 只有进入 `RECOMMENDED` 的论文才允许进入 PDF 下载环节
- 只有进入 `DRAFT_READY` 的草稿才允许进入审核
- 只有审核通过的草稿才允许发起发布
- 已发布内容默认不可重复发布，除非显式创建新的发布任务

## 审核流

建议最小审核流：

1. 系统生成草稿
2. 审核人打开草稿
3. 审核人修改标题/摘要/正文
4. 审核通过或驳回
5. 审核通过后才能发布

## 重试策略

- API 调用失败：自动重试
- 长耗时任务失败：进入待人工干预队列
- 发布失败：允许人工重试
- 重试行为需要带上原始上下文和错误摘要

## 审计要求

以下行为必须可回溯：

- 状态变更
- 审核通过/驳回
- 手动重试
- 发布动作

## 与代码生成的关系

本文件会直接影响：

- 数据库状态字段设计
- 任务调度器实现
- 后端状态流转校验
- 前端审核页和发布页行为

## 需要你补充的信息

- 审核是否必须双人复核
- 驳回后是否允许重新生成草稿
- 发布是否需要“预约发布时间”
- 发布失败后是否允许自动降级为草稿保存
